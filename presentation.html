01 · The Problem

Models edit well. They don't maintain well.

✓ What models do well

  • Apply a single rule change cleanly
  • Parse and re-emit structured YAML
  • Understand natural-language policy memos
  • Make isolated edits accurately (~85% correct per update)

✗ What models fail at

  • Tracking cumulative state across 8–10 interacting changes
  • Re-computing global priority order after mid-stream adds + revocations
  • Recognising that a revocation reshapes which remaining rules now compete
  • Submitted policy wrong in 40–80% of scenarios after 3 swaps + 2 revocations

The failure is not editing ability. It is the inability to mentally execute a sequence of non-commutative mutations on a shared data structure and arrive at the correct final state. This matters in real-world policy maintenance: access control, compliance rules, pricing logic, SLA escalation trees.

02 · The Task

What the model receives

Initial policy (YAML)

definitions:
  standard_window: 60      # shared numeric constant
  max_refund_amount: 100

rules:
- id: r001
  conditions: []           # empty = catch-all default
  decision: request_documentation
  priority: 1

- id: r005
  conditions:
  - customer_tier in ['platinum']
  - item_condition == 'defective'
  decision: approve_refund
  priority: 5

- id: r012
  conditions:
  - purchase_channel == 'online'
  decision: escalate_to_manager
  priority: 12
# ... 9 more rules

Key: highest priority wins when multiple rules match. Definitions are shared — changing one propagates to all rules that reference it.

Update stream (8–10 memos, all in the same surface form)

  • 1
    Situations in which the customer tier is platinum are to be handled as: escalate to a manager.
  • 2
    Where the purchase channel is phone, and the final sale status is active, and the item price is at most 100 and the customer tier is platinum, and the item condition is defective overlap, offer store credit is the applicable outcome.
  • 3
    The separate handling for cases where the customer tier is platinum, and the item condition is defective has been removed.
  • 4
    Where the customer tier is platinum applies, the outcome is deny the refund.
  • 5
    When the purchase age (in days) is less than 100, and the purchase channel is online or in store, and the customer tier is silver and the purchase channel is phone or in store both hold, the outcome is request documentation.
  • 6
    No distinct rule governs cases where the final sale status is not active, and the customer tier is platinum or silver.
  • 7
    Cases satisfying both [purchase age < 100, channel in online/in_store, tier = silver] and [channel = online, condition = defective, age > 60] result in request documentation, not approve the refund.
  • 8
    Cases where the purchase channel is phone or in store should result in offer store credit.

Critical design choice: all update types — adds, revocations, priority swaps, no-ops — use the same wording. The model cannot infer the operation type from the memo text.

The 6 Update Types — What Can Appear in the Stream

Every memo is one of six types. The model never sees which.

All six types use identical surface form: "When [conditions], [decision]." The model must reason from the semantics alone — not from wording — to figure out what YAML change (if any) is required.

chain_create — Add a new rule

What it means: a brand new rule needs to be inserted into the policy with new conditions, a decision, and a priority.

Correct action: add a new rule block to the YAML. Assign it a priority that fits above or below existing rules correctly.

"Situations in which the customer tier is platinum are to be handled as: escalate to a manager."

Trap: the model must not duplicate an existing rule that already covers platinum customers.

chain_ref — Modify an existing rule

What it means: a rule that was already created (often by a prior chain_create in the same stream) needs its decision changed.

Correct action: find the existing rule with those conditions and update only its decision field. Do not add a new rule.

"Where the customer tier is platinum applies, the outcome is deny the refund." (same conditions as the chain_create — different decision)

Trap: looks identical to chain_create. Model must recognise the conditions already exist and modify rather than add.

change_priority — Priority swap

What it means: two rules whose conditions overlap are in the wrong priority order. The memo names the conditions of both and states which decision should win.

Correct action: adjust the priority numbers so the winning rule's priority is higher than the losing rule's. All other rules must remain correctly ordered relative to each other.

"Where [condition A] and [condition B] both hold, offer store credit is the applicable outcome."

Trap: fixing one pair can break the relative order of other rules. Three of these in a row is what causes priority arithmetic collapse.

revoke_rule — Delete a rule entirely

What it means: a rule that currently exists in the policy should be removed completely. No separate handling for those conditions anymore.

Correct action: delete the entire rule block from the YAML. Do not modify it — remove it.

"The separate handling for cases where the customer tier is platinum and the item condition is defective has been removed."

Trap: surface form is almost identical to a chain_ref. Model must detect "removed" / "no longer" language and delete, not modify.

no_op_disguised — Do nothing

What it means: the memo describes behavior that is already correctly handled by an existing rule. No change to the YAML is needed.

Correct action: recognise that this scenario is already covered. Make zero edits.

"Cases where the purchase channel is phone or in store should result in offer store credit." (a rule for this already exists at priority 10)

Trap: looks like every other memo. Adding a redundant rule creates a priority conflict or shadow rule that breaks other scenarios.

update_default — Change the fallback

What it means: the catch-all default rule (empty conditions, always priority 1) needs its decision changed.

Correct action: find the rule with empty conditions and update its decision field only.

"Unmatched cases result in escalate to a manager."

Trap: only appears in 10-update tasks. Model must identify the default rule (empty conditions) rather than creating a new rule.

The core difficulty: all six types arrive in the same uniform memo format. There are no labels, no keywords that reliably signal which type it is. The model must infer the operation from the semantics — and do this correctly for all 8–10 memos simultaneously before making any edits.

03 · Tool Design — No Oracle

5 tools. Zero behavioral feedback before submission.

What the model has

ToolReturnsWhat it reveals
read_policy()Current YAMLFull artifact text
read_updates()All memosFull update stream
edit_policy(yaml){"status":"ok"}Nothing — blind replace
compile_check()ok or errorSyntax only — no behavior
submit()Final scoreOnly after submission

edit_policy() is deliberately blind. It returns only {"status":"ok"} — no rule count, no priority summary, no structural hints. The model cannot sanity-check its edit before submitting.

What is deliberately absent

Behavioral preview

No tool lets the model check whether its edited policy is behaviorally correct before submitting. The model must reason its way to the answer — it cannot iterate toward it using feedback.

Structural summary

No tool returns rule IDs, rule counts, or a priority list. The model must track structural state entirely from its own reads of the policy YAML.

Error localisation

No tool pinpoints which scenario type is wrong. Submission is the only moment of truth — and it's irreversible.

04 · Scoring — How We Measure Correctness

Behavioral equivalence: 2000 scenarios, 95% threshold

How a task is scored

1
Compile both policies
The model's submitted YAML and the hidden gold YAML are both parsed into executable decision functions by the same compiler.
2
Sample 2,000 scenarios
Random combinations of all domain variables (customer_tier, purchase_channel, item_condition, etc.) are drawn from a fixed seed. Same 2,000 scenarios every time.
3
Compare decisions
Each scenario is evaluated against both policies. The score is the fraction where they agree: score = matches / 2000.
4
Pass/fail threshold: 95%
A task passes if score ≥ 0.95 (at least 1,900 of 2,000 scenarios agree). Fail = score < 0.95.

Why these specific numbers?

Why 2,000 scenarios?

The variable space for each domain has thousands of possible combinations. 2,000 samples gives statistical power to detect even moderate error rates (a policy wrong on 5% of cases will fail ~100 out of 2,000 checks). It is large enough to be reliable and small enough to run in under a second. The seed is fixed so results are fully reproducible.

Why 95% — not 75%, 85%, or 90%?

The key observation: a wrong priority swap doesn't fail one scenario — it fails every scenario where both conflicting rules match. That's a region, not a point. We looked at every failed task in our results and measured how many mismatches each wrong-priority error actually produced:

Smallest error observed
~200 mismatches = 10% of scenarios
Largest error observed
1,326 mismatches = 66% of scenarios
Typical error
400–700 mismatches = 20–35% of scenarios

This directly sets the threshold. Even the gentlest real priority error drops the score to ~0.90. So:

  • 75% / 85% — a model could miss 2–3 priority swaps and still pass. Not measuring global coherence at all.
  • 90% — a model that misses exactly one "small" priority swap (10% overlap) scores exactly 0.90 and passes. The benchmark misses the failure it was designed to catch.
  • 95% — any real priority error (minimum 10% mismatches → score ≤ 0.90) guarantees failure. Meanwhile, 95% still forgives genuine ambiguity: two non-interacting rules that could be assigned priority 8 or 9 interchangeably are behaviorally identical and don't affect the score.

95% is the exact crossing point where "passed" and "got every meaningful swap right" become the same statement. One meaningful error → guaranteed fail. Genuine equivalent orderings → still pass.

No LLM judge. No rubric. No partial credit. The score is exact and deterministic — given any two policies and any seed, the score is always the same. This makes results fully reproducible and comparable across models.

Where does the gold policy come from?

The gold is not labeled by a human. Our Python code applies each update directly to the rule graph (e.g. swap two priority integers, delete a rule). The gold is the result of calling graph.to_yaml() on the final mutated Python object. It's mechanically derived — then verified by scoring it against itself (must achieve 2000/2000) before any task is accepted.

05 · How Test Cases Are Generated

Each task is built from a latent rule graph the model never sees

Generation pipeline

1
Pick a domain
One of 4 domains: refund, approval, access_control, escalation. Each has its own variables, decision types, and definition templates.
2
Build the initial rule graph (12–16 rules)
Random conditions over domain variables, random decisions, 40% chance of override relationships. Numeric conditions reference shared definitions.
3
Python mutates the graph directly
For each update in the skeleton, Python code mutates the rule graph first — e.g. rule_a["priority"], rule_b["priority"] = rule_b["priority"], rule_a["priority"] for a swap, or graph.rules.pop(id) for a revocation. Then a natural-language memo is generated as a description of what just happened. The sentences are outputs, not inputs.
4
Memos sent to model. Graph stays hidden.
The agent only receives the initial YAML and the memo sentences. It must figure out what YAML changes the memos imply — the model is doing the hard part. The latent graph (ground truth) is never shown.
5
Gold policy = graph.to_yaml() after all mutations
The gold is produced by calling to_yaml() on the final mutated Python object. No human labeled it. It is verified by running it through the same scorer — gold must score 2000/2000 against itself before the task is accepted.

Quality filters applied to every task

  • Behavioral gap filter — skip if the initial (uneditied) policy already scores ≥ 85% against gold. The task must require meaningful changes.
  • Gold self-check — verify gold achieves 2000/2000 against itself. Catches generator bugs.
  • Ambiguity filter — reject any memo containing vague words ("generally", "usually", "where appropriate"). All descriptions must be unambiguous.
  • Uniform surface form — all update types produce the same memo format. The model cannot identify add vs. revoke vs. swap from wording.

06 · Failure Modes — How Models Break

3 precise, mechanistic failure categories

Every mismatch between the submitted policy and the gold policy is classified into one of these three categories. This is not a rubric — it is derived mechanically by comparing rule structures and identifying why the decision differed for each failing scenario.

wrong_priority

What it is: A scenario was matched by multiple rules. The wrong rule won because its priority number was incorrect in the submitted policy.

Root cause: The model failed to track how 3 priority swaps — applied after mid-stream adds and revocations — reshaped the global ranking.

Most common. Appears in 100% of failed tasks. When you revoke 2 rules and then swap 3 pairs, the final ordering is completely different from what the model computed.

Example: task_0000 — 591 scenarios wrong because priority arithmetic collapsed after 3 swaps.
stale_rule_retained

What it is: A rule that was revoked by an update still appears in the submitted policy. It matches scenarios it should no longer cover, producing wrong decisions.

Root cause: The uniform memo surface form makes revocations look identical to modifications. The model fails to recognise which updates are deletions.

Second most common. Appears in ~80% of failed tasks. A retained stale rule typically has high priority, making it dominate large portions of the scenario space.

Example: task_0001 — 518 scenarios wrong because one revoked rule was kept with priority 7.
wrong_decision

What it is: A rule has the correct conditions and correct priority — but its decision field is wrong. The model identified the right rule but applied the wrong outcome.

Root cause: Usually a missed chain_ref — an update that changed the decision of an earlier rule (chain dependency tracking failure).

Least common. Models are generally good at tracking decision changes. Appears in ~15% of failed tasks, usually alongside priority errors.

Example: task_0010 — 62 scenarios wrong because update 9 (second chain_ref) changed a decision that the model missed.

Prompting Architecture — How the API Calls Work

One prompt with everything. Then a tool-call loop until submit.

A common question: does the model get the initial policy in one prompt and then updates one-by-one? No. Here is the exact structure.

Step 1 — One system prompt (same for every task)

You are a policy maintenance agent.
Your task is to update a YAML policy
to correctly reflect a set of changes.

CRITICAL: You have NO behavioral
feedback before submission.
compile_check() tells you only whether
YAML parses — no behavioral hints.
edit_policy() returns only {"status":"ok"}.

Think globally. Submit when confident.

This is the model's "briefing." It explains the DSL semantics, the no-oracle constraint, and the reasoning discipline. It is sent once per conversation as the system prompt.

Step 2 — One user message with full context

## Domain
refund

## Your Task
Modify the policy so it correctly
reflects ALL updates.

## Current Policy
```yaml

  standard_window: 60
  ...

- id: r001

  decision: request_documentation
  priority: 1
... (all 12–16 rules) ...
```

## Updates to Apply
1. Situations in which the customer
   tier is platinum are to be handled
   as: escalate to a manager.
2. Where the purchase channel is
   phone ...
... (all 8–10 updates) ...

All updates are listed together in one message — not streamed one at a time. The model must process all 8–10 memos simultaneously and reason about their cumulative effect.

Step 3 — Tool-call loop (model-driven)

→
Model calls read_policy()
Gets back the full YAML text (same as the prompt — serves as a re-read)
→
Model calls read_updates()
Gets back the list of update strings (same as the prompt)
→
Model calls edit_policy(new_yaml)
Returns {"status":"ok"} — no feedback on correctness
→
Model calls compile_check()
Returns ok or syntax error — nothing about behavior
→
Model calls submit()
Score revealed. Conversation ends.

The model drives the loop. It decides which tools to call, in what order, and when to submit. Most successful runs used exactly 5 calls: read_policy → read_updates → edit_policy → compile_check → submit. Failed runs sometimes skipped the edit or made multiple edits without behavioral feedback to guide them.

Why give read_policy + read_updates as tools if the content is already in the prompt?

Two reasons: (1) it gives the model an explicit "re-read" action that some models use to re-anchor their context before editing; (2) it creates a realistic tool-use environment — in production, an agent would not have the policy pre-loaded but would fetch it. The prompt version is the agent's initial context; the tool version is the canonical source during the conversation.

Prototype Implementation — What Was Built

6 runnable files. All independently testable.

Failure trajectories are the most valuable artifact for training. They show exactly where the model's reasoning broke down — which edit was wrong, how wrong, and in what way. This enables targeted RLHF or fine-tuning on the specific failure modes.

Why StateBench Is Useful

For evaluation and for training

For evaluation

  • Clean capability gap: global state tracking under sequential interacting updates — not hallucination, formatting, or refusal
  • Hard to overfit: combinatorial task space, exact behavioral equivalence scoring (no rubric), surface form uninformative about update type
  • Maps to production tasks: config maintenance, rule engine updates, sequential migrations, compliance document edits
  • No LLM judge: fully deterministic, identical score on every run with the same policies and seed
  • Tunable difficulty: rule count, update count, skeleton type — all independently controllable

For training

  • Precise failure taxonomy: wrong_priority / stale_rule_retained / wrong_decision each pinpoints where the model lost global state
  • Rich failure trajectories: each failure shows which priority swap was wrong, which rule was kept — enabling targeted fine-tuning
  • Correct trajectories as positive signal: show the desired reasoning pattern — simulate all mutations, then edit once
  • Scales to thousands of tasks: 4 domains, controllable difficulty, infinite seed space — suitable as both eval suite and training data generator

The benchmark can generate training data at scale for the exact failure mode it measures: priority arithmetic across non-commutative mutations on a shared rule system.
